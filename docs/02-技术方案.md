# 五子棋机器人 — 技术方案

> 版本: V0.3 | 日期: 2026-03-30

## 1. 方案选型

### 1.1 总体架构: 混合方案

**混合方案的分工：**

| 模块 | 方案 | 为什么 |
|------|------|--------|
| 棋盘识别 | ArUco + HSV 阈值 | 结构化场景，不需要训练 |
| 落子决策 | α-β 搜索 + 启发式评分 | 确定性强，9×9 深度 4 已足够 |
| 操控执行 | **Pi0 VLA** (语言→动作) | 从语言指令生成关节轨迹，泛化性好 |
| 棋子识别 | 不需要 | HSV 已解决，棋子在有序托盘中 |
| 语音对话 | **LLM + TTS/ASR** | 有性格的对话，不是模板 |
| 情绪动作 | **动作表演库 + 状态机** | 机械臂"微表情"是核心差异化 |

### 1.2 各模块方案选型

#### 视觉: 传统 CV（无需训练）

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **传统 CV (ArUco + HSV)** | 无需训练、<10ms、可解释 | 对光照敏感 | **采用** |
| YOLO 检测 | 鲁棒性好 | 需标注数据、过于重量级 | 不需要 |

ArUco 标记贴在棋盘角落（1 个），用于定位棋盘在图像中的位置。然后根据已知的棋盘尺寸推算 81 个格点的像素坐标，逐个检测颜色。

#### 棋类引擎: α-β 搜索（无需训练）

对于 9×9 棋盘，depth 4 的 α-β 搜索已经足够强。搜索优化：
- 邻域裁剪：只搜索已有棋子周围 2 格内的空位
- 启发式排序：按威胁模式评分排序候选位置
- 搜索宽度限制 10-15 个候选

#### 操控策略: Pi0 VLA

| 方案 | 参数量 | 输入 | 训练数据 | 硬件要求 |
|------|--------|------|---------|---------|
| IK 脚本 | 0 | 坐标 | 0 | 任意 |
| ACT | ~10M | 图像+关节+坐标 | 500+ demos | 8GB GPU |
| **Pi0 VLA** | ~3B | 图像+关节+**语言** | 1000 demos | **Spark 128GB** |
| SmolVLA | ~1B | 图像+关节+语言 | 500+ demos | 16GB GPU |

**选择 Pi0 的理由：**
- 语言条件化：目标位置通过自然语言传达，更灵活
- 预训练视觉理解：SigLIP 编码器已理解图像，微调数据需求少
- 硬件支持：NVIDIA Spark 128GB 可全精度训练
- LeRobot 原生集成

**IK 脚本作为 fallback：** 如果 Pi0 训练效果不理想，可退回到脚本化 IK 控制（已在仿真中验证通过）。

## 2. Pi0 VLA 策略设计

### 2.1 架构

```
输入:
  图像 ──────→ SigLIP 视觉编码器 ──→ 视觉 token
  语言指令 ──→ Gemma 语言模型 ──────→ 语言 token   ──→ Gemma Decoder ──→ 动作序列
  关节角度 ──→ 线性投影 ────────────→ 状态 token                          (50帧 × 6D)

输出:
  未来 50 帧的关节轨迹 (50 × 6D)  ← "action chunking"
```

### 2.2 语言指令设计

每个 episode 的语言指令支持多种表达（数据增强）：

```python
templates = [
    "把{color}棋放到第{row}行第{col}列",
    "Place {color} stone at row {row}, column {col}",
    "落子 ({row}, {col})",
    "{color} to {row},{col}",
]
```

### 2.3 训练数据格式

LeRobot dataset 格式，每帧包含：

| 字段 | 类型 | 形状 | 说明 |
|------|------|------|------|
| `observation.images.top` | video | (240, 320, 3) | 俯视相机 RGB |
| `observation.state` | float32 | (6,) | 当前关节角度 |
| `action` | float32 | (6,) | 关节角度目标（专家动作） |
| `language_instruction` | string | — | 自然语言指令 |

### 2.4 训练配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 预训练权重 | lerobot/pi0-base | PaliGemma 基础 |
| 微调方式 | 全精度 (Spark 128GB) | 不需要 LoRA |
| batch_size | 8 | |
| 训练步数 | 50K | ~8-16h on Spark |
| learning_rate | 1e-4 | |
| chunk_size | 50 | ~1.7s @ 30fps |
| n_action_steps | 25 | 执行 25 步后重新推理 |

### 2.5 Domain Randomization

| 类别 | 随机化内容 | 范围 |
|------|-----------|------|
| 棋子位置 | 托盘上 XY 微扰 | ±5mm |
| 目标位置 | 9×9 格点均匀采样 | 全部 81 个点 |
| 光照 | 强度 + 色温 | ±30%, 4000-6500K |
| 相机位姿 | XYZ + RPY | ±1cm, ±2° |
| 关节噪声 | 动作执行噪声 | ±0.3° |

## 3. 情绪动作系统

机械臂的动作就是表情——不需要屏幕，6-DOF 关节本身就能表达情绪。这是产品核心差异化。

### 3.1 情绪状态机

```
┌─────────────────────────────────────────────────┐
│            情绪状态机 (EmotionStateMachine)        │
│                                                   │
│  棋类引擎输出 ──→ 情绪判定 ──→ 动作选择 ──→ 轨迹  │
│                                                   │
│  输入:                                            │
│    - AI 评估分数 (前 N 个候选)                     │
│    - 局势变化 (优势/劣势/逆转)                     │
│    - 对手落子质量                                  │
│    - 连胜/连败状态                                 │
│                                                   │
│  输出:                                            │
│    - 情绪标签 (果断/犹豫/紧张/得意/失落/惊讶)       │
│    - 动作参数 (速度倍率/抖动幅度/停顿时长)          │
│    - 语音触发信号                                  │
└─────────────────────────────────────────────────┘
```

### 3.2 情绪判定规则

| 触发条件 | 情绪 | 判定依据 |
|----------|------|---------|
| 有明显最优解 | 果断 | top1 分数 >> top2 |
| 前两个候选分数接近 | 犹豫 | \|score1 - score2\| < threshold |
| 局势突然不利 | 紧张 | 评估分骤降 |
| 走出致胜步 | 得意 | 检测到必胜局面 |
| 输棋 | 失落 | game over, AI 输 |
| 对手下了好棋 | 惊讶 | 对手落子后 AI 评估分骤降 |
| 低概率随机 | 手滑 | random() < 0.03 |

### 3.3 动作表演库

每个情绪对应一组参数化的动作原语，叠加在 Pi0 基础轨迹上：

| 动作 | 实现方式 | 参数 |
|------|---------|------|
| 犹豫 | 手在两个候选位置间来回摆动 | 摆动次数 2-4，幅度 ±20mm |
| 果断 | 加速直线运动，干脆落子 | 速度 ×1.5，无停顿 |
| 手抖 | 关节叠加高频正弦抖动 | 频率 8-12Hz，幅度 ±0.5° |
| 掉棋子 | 夹爪提前松开 → 暂停 → 重新夹起 | 触发概率 3% |
| 拍桌 | 夹爪闭合，手臂快速下压桌面 | 力度可配置 |
| 垂手 | 手臂缓慢下垂到休息姿态 | 速度 ×0.3 |
| 得意 | 落子后手臂高举 | 抬升 15cm，停留 1s |
| 偷看 | 手臂探向对手方向又缩回 | 等待对手时随机触发 |

### 3.4 实现方案

- **IK 脚本模式：** 动作表演用预定义的关节轨迹实现，参数化控制速度/幅度/停顿
- **Pi0 模式：** 基础抓放由 Pi0 完成，表演动作在 Pi0 轨迹前后拼接
- 动作原语存储为 JSON 配置，支持运行时调整

## 4. 语音交互系统

### 4.1 架构

```
麦克风 → ASR (语音识别) → 文本 → LLM (对话生成) → 文本 → TTS (语音合成) → 扬声器
                                      ↑
                              情绪状态 + 棋局上下文
```

### 4.2 方案选型

| 模块 | 方案 | 备选 | 说明 |
|------|------|------|------|
| ASR | Whisper (本地) | 云端 API | 离线优先，延迟 <1s |
| LLM | 云端大模型 API | 本地小模型 | 生成有性格的对话，非模板 |
| TTS | Edge-TTS / 本地 | 云端 API | 低延迟，可选音色 |

### 4.3 性格 Prompt 设计

LLM 对话以系统 prompt 注入棋局状态和性格设定：

```
你是一个五子棋机器人。你的性格是{personality}。
当前局势: {game_state_summary}
你的情绪: {emotion}
最近事件: {event}  (例: "对手刚下了一步好棋威胁到你")

请生成一句简短的口语化回应（<15字）。
```

性格模式（P1 功能）：
- **话痨：** 每步都说话，喜欢评论
- **安静：** 只在关键时刻说话
- **毒舌：** 嘲讽对手
- **鼓励：** 夸对手，适合新手

### 4.4 语音触发时机

| 时机 | 示例 | 是否阻塞 |
|------|------|---------|
| 开局 | "我先来。" | 否 |
| 思考中 | "嗯...让我想想..." | 否，与思考并行 |
| 落子后 | "差点没看到！" | 否 |
| 等待对手 | "你确定不悔棋？" | 否 |
| 赢/输 | "再来一局？" | 是，等语音播完 |
| 对手语音指令 | ASR → 解析 → 执行 | 是 |

## 5. 难度系统

### 5.1 三档难度

| 难度 | 搜索深度 | 候选宽度 | 策略调整 | 用户感知 |
|------|---------|---------|---------|---------|
| 简单 | 2 | 8 | 偶尔故意忽略最优解 (20%) | "让着你" |
| 中等 | 4 | 12 | 正常搜索 | 有挑战性 |
| 困难 | 6 | 15 | 完全最优 + 开局库 | 很难赢 |

### 5.2 切换方式

- 语音："简单模式" / "困难模式"
- 自动建议：连续赢/输 3 局后提示调整

```
│  │  逐帧发送给 SO-100 电机                    │           │
│  └───────────────────────────────────────────┘           │
│       ↓                                                  │
│  [表演后动作] → (得意举手 / 拍桌 / ...)                    │
│       ↓                                                  │
│  拍照验证落子 → 切换颜色 → 等待人类下棋                    │
│       ↓                                                  │
│  [等待中] → ASR 监听语音指令 ("悔棋" / "简单模式")         │

│                                                          │
└──────────────────────────────────────────────────────────┘
```

### 7.3 坐标系与标定

三个坐标系：像素 → 棋盘 → 机器人

| 步骤 | 方法 | 输入 | 输出 |
|------|------|------|------|
| 像素→棋盘 | ArUco + 单应性矩阵 | 4 个角点像素坐标 | H (3×3) |
| 棋盘→机器人 | 3D 刚体变换 (SVD) | 4 个点的 EE 坐标 | T (4×4) |
| 托盘标定 | 两点定义直线 | 首末槽位 EE 坐标 | 原点 + 方向向量 |

## 8. 开箱即用方案

需求要求 3 分钟内开始第一局，不需要 App、WiFi、手动校准。

### 8.1 自动标定流程

```
开机 → 检测 ArUco 标记 → 自动计算棋盘位姿 →
检测棋盒位置 (预定义相对位置) → 就绪
```

- ArUco 标记贴在棋盘角落，用户只需贴一次
- 棋盒放在机械臂旁固定位置（底座有标记线引导）
- 像素→棋盘映射在每局开始时自动更新（光照变化鲁棒）
- 棋盘→机器人映射出厂预标定，用户无需操作

### 8.2 交互启动

```
插电 → 机械臂自检 (归零) → 相机检测棋盘 →
```

## 10. Sim-to-Real 迁移方案

```
仿真训练完成后:

Step 1: 零样本测试 (zero-shot)
  直接加载仿真训练的 Pi0 → 真实 SO-100 → 测试成功率

Step 2: 评估差距
  > 50% → 直接微调
  30-50% → 需要一些真实数据
  < 30% → 需要加强 domain randomization

Step 3: 真实数据微调 (如需要)
  用 Leader 臂示教 20-50 条 → 混合仿真数据微调 → Spark 上训练

Step 4: 部署
  Pi0 策略接入 main.py → 完整对弈循环
```

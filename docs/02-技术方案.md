# 五子棋机器人 — 技术方案

> 版本: V0.2 | 日期: 2026-03-27

## 1. 方案选型

### 1.1 总体架构: 混合方案

```
┌────────────────────────────────────────────────────────┐
│                     主控 (PC)                           │
│                                                        │
│  ┌──────────┐   ┌──────────┐   ┌───────────────────┐  │
│  │ 视觉模块  │──▶│ 棋类引擎  │──▶│   操控模块          │  │
│  │ ArUco+HSV │   │ α-β搜索  │   │  Pi0 VLA 策略     │  │
│  │ (棋盘识别) │   │ (落子决策) │   │  (语言→动作)      │  │
│  └────┬─────┘   └─────┬────┘   └─────────┬─────────┘  │
│       │               │                   │            │
│  ┌────┴─────┐   ┌─────┴─────┐   ┌────────┴────────┐   │
│  │ USB 相机  │   │ 语言指令   │   │  SO-100 机械臂   │   │
│  │ 640×480  │   │生成器      │   │  Feetech STS3215│   │
│  └──────────┘   └───────────┘   └─────────────────┘   │
│                                                        │
│  ┌─────────────────────────────────────────────────┐   │
│  │              协调器 (Orchestrator)                │   │
│  │  game loop / 状态机 / 错误恢复 / 日志            │   │
│  └─────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────┘
```

**混合方案的分工：**

| 模块 | 方案 | 为什么 |
|------|------|--------|
| 棋盘识别 | ArUco + HSV 阈值 | 结构化场景，不需要训练 |
| 落子决策 | α-β 搜索 + 启发式评分 | 确定性强，9×9 深度 4 已足够 |
| 操控执行 | **Pi0 VLA** (语言→动作) | 从语言指令生成关节轨迹，泛化性好 |
| 棋子识别 | 不需要 | HSV 已解决，棋子在有序托盘中 |

### 1.2 各模块方案选型

#### 视觉: 传统 CV（无需训练）

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **传统 CV (ArUco + HSV)** | 无需训练、<10ms、可解释 | 对光照敏感 | **采用** |
| YOLO 检测 | 鲁棒性好 | 需标注数据、过于重量级 | 不需要 |

ArUco 标记贴在棋盘角落（1 个），用于定位棋盘在图像中的位置。然后根据已知的棋盘尺寸推算 81 个格点的像素坐标，逐个检测颜色。

#### 棋类引擎: α-β 搜索（无需训练）

对于 9×9 棋盘，depth 4 的 α-β 搜索已经足够强。搜索优化：
- 邻域裁剪：只搜索已有棋子周围 2 格内的空位
- 启发式排序：按威胁模式评分排序候选位置
- 搜索宽度限制 10-15 个候选

#### 操控策略: Pi0 VLA

| 方案 | 参数量 | 输入 | 训练数据 | 硬件要求 |
|------|--------|------|---------|---------|
| IK 脚本 | 0 | 坐标 | 0 | 任意 |
| ACT | ~10M | 图像+关节+坐标 | 500+ demos | 8GB GPU |
| **Pi0 VLA** | ~3B | 图像+关节+**语言** | 1000 demos | **Spark 128GB** |
| SmolVLA | ~1B | 图像+关节+语言 | 500+ demos | 16GB GPU |

**选择 Pi0 的理由：**
- 语言条件化：目标位置通过自然语言传达，更灵活
- 预训练视觉理解：SigLIP 编码器已理解图像，微调数据需求少
- 硬件支持：NVIDIA Spark 128GB 可全精度训练
- LeRobot 原生集成

**IK 脚本作为 fallback：** 如果 Pi0 训练效果不理想，可退回到脚本化 IK 控制（已在仿真中验证通过）。

## 2. Pi0 VLA 策略设计

### 2.1 架构

```
输入:
  图像 ──────→ SigLIP 视觉编码器 ──→ 视觉 token
  语言指令 ──→ Gemma 语言模型 ──────→ 语言 token   ──→ Gemma Decoder ──→ 动作序列
  关节角度 ──→ 线性投影 ────────────→ 状态 token                          (50帧 × 6D)

输出:
  未来 50 帧的关节轨迹 (50 × 6D)  ← "action chunking"
```

### 2.2 语言指令设计

每个 episode 的语言指令支持多种表达（数据增强）：

```python
templates = [
    "把{color}棋放到第{row}行第{col}列",
    "Place {color} stone at row {row}, column {col}",
    "落子 ({row}, {col})",
    "{color} to {row},{col}",
]
```

### 2.3 训练数据格式

LeRobot dataset 格式，每帧包含：

| 字段 | 类型 | 形状 | 说明 |
|------|------|------|------|
| `observation.images.top` | video | (240, 320, 3) | 俯视相机 RGB |
| `observation.state` | float32 | (6,) | 当前关节角度 |
| `action` | float32 | (6,) | 关节角度目标（专家动作） |
| `language_instruction` | string | — | 自然语言指令 |

### 2.4 训练配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 预训练权重 | lerobot/pi0-base | PaliGemma 基础 |
| 微调方式 | 全精度 (Spark 128GB) | 不需要 LoRA |
| batch_size | 8 | |
| 训练步数 | 50K | ~8-16h on Spark |
| learning_rate | 1e-4 | |
| chunk_size | 50 | ~1.7s @ 30fps |
| n_action_steps | 25 | 执行 25 步后重新推理 |

### 2.5 Domain Randomization

| 类别 | 随机化内容 | 范围 |
|------|-----------|------|
| 棋子位置 | 托盘上 XY 微扰 | ±5mm |
| 目标位置 | 9×9 格点均匀采样 | 全部 81 个点 |
| 光照 | 强度 + 色温 | ±30%, 4000-6500K |
| 相机位姿 | XYZ + RPY | ±1cm, ±2° |
| 关节噪声 | 动作执行噪声 | ±0.3° |

### 4.2 主循环数据流

```
┌─ 外层循环 (每步棋) ─────────────────────────────┐
│                                                  │
│  相机拍照 → ArUco定位 → HSV检测 → 棋盘状态        │
│       ↓                                          │
│  Alpha-Beta AI → 最佳着法 (row, col)              │
│       ↓                                          │
│  生成语言指令: "把黑棋放到第{row}行第{col}列"       │
│       ↓                                          │
│  ┌─ Pi0 执行 (~1.7s) ───────────────────────┐    │
│  │  图像+关节+语言 → Pi0 → 50帧关节轨迹      │    │
│  │  逐帧发送给 SO-100 电机                    │    │
│  └───────────────────────────────────────────┘    │
│       ↓                                          │
│  拍照验证落子 → 切换颜色 → 等待人类下棋            │
│                                                  │
└──────────────────────────────────────────────────┘
```

### 4.3 坐标系与标定

三个坐标系：像素 → 棋盘 → 机器人

| 步骤 | 方法 | 输入 | 输出 |
|------|------|------|------|
| 像素→棋盘 | ArUco + 单应性矩阵 | 4 个角点像素坐标 | H (3×3) |
| 棋盘→机器人 | 3D 刚体变换 (SVD) | 4 个点的 EE 坐标 | T (4×4) |
| 托盘标定 | 两点定义直线 | 首末槽位 EE 坐标 | 原点 + 方向向量 |

## 6. Sim-to-Real 迁移方案

```
仿真训练完成后:

Step 1: 零样本测试 (zero-shot)
  直接加载仿真训练的 Pi0 → 真实 SO-100 → 测试成功率

Step 2: 评估差距
  > 50% → 直接微调
  30-50% → 需要一些真实数据
  < 30% → 需要加强 domain randomization

Step 3: 真实数据微调 (如需要)
  用 Leader 臂示教 20-50 条 → 混合仿真数据微调 → Spark 上训练

Step 4: 部署
  Pi0 策略接入 main.py → 完整对弈循环
```

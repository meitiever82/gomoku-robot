# 五子棋机器人 — 技术方案

> 版本: V0.1 | 日期: 2026-03-19

## 1. 方案选型

### 1.1 总体架构: 混合方案

```
┌────────────────────────────────────────────────────────┐
│                     主控 (PC / Orin)                    │
│                                                        │
│  ┌──────────┐   ┌──────────┐   ┌───────────────────┐  │
│  │ 视觉模块  │──▶│ 棋类引擎  │──▶│   操控模块          │  │
│  │ OpenCV   │   │ α-β搜索  │   │  ACT Policy       │  │
│  │ ArUco    │   │ 威胁评估  │   │  + IK fallback    │  │
│  └────┬─────┘   └──────────┘   └─────────┬─────────┘  │
│       │                                    │            │
│  ┌────┴─────┐                    ┌────────┴────────┐   │
│  │ USB 相机  │                    │  SO-100 机械臂   │   │
│  │ 640×480  │                    │  Feetech STS3215│   │
│  └──────────┘                    └─────────────────┘   │
│                                                        │
│  ┌─────────────────────────────────────────────────┐   │
│  │              协调器 (Orchestrator)                │   │
│  │  game loop / 状态机 / 错误恢复 / 日志            │   │
│  └─────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────┘
```

**为什么选混合方案而非端到端学习：**

| 维度 | 端到端学习 | 混合方案 | 选择理由 |
|------|----------|---------|---------|
| 下棋决策 | 需要海量棋局数据训练 | 传统搜索算法，确定性强 | 五子棋 AI 算法成熟，无需学习 |
| 操控执行 | 策略同时学会"看"和"做" | 专注学习抓放动作 | 任务分解降低学习难度 |
| 数据需求 | 数千局完整对弈 | 30-50 次抓放示教 | 数据采集成本低一个数量级 |
| 可调试性 | 黑盒 | 每个模块可独立测试 | 出问题容易定位 |
| 可靠性 | 不确定 | 决策层确定性保证 | 棋不能下错 |

### 1.2 各模块方案选型

#### 视觉: 传统 CV vs 深度学习

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **传统 CV (ArUco + HSV)** | 无需训练、<10ms、可解释 | 对光照敏感 | **采用** |
| YOLO 检测 | 鲁棒性好 | 需标注数据、过于重量级 | 备选 |
| 模板匹配 | 简单 | 不抗旋转 | 不适合 |

**传统 CV 的合理性：** 棋盘是高度结构化场景——规则网格、已知尺寸、高对比度棋子。ArUco 标记解决定位问题，HSV 阈值解决分类问题。用深度学习杀鸡用牛刀。

**光照敏感性应对：**
- 使用 LED 环形灯/漫射罩提供稳定照明
- 自适应阈值：开局时采样空棋盘的 HSV 基准值
- 多帧平均减少噪声

#### 棋类引擎: 搜索深度与算法

| 方案 | 棋力 | 耗时 (9×9) | 实现复杂度 |
|------|------|-----------|-----------|
| 随机落子 | 极弱 | <1ms | 极低 |
| 启发式评分 | 中等 | <10ms | 低 |
| **α-β + 威胁评估** | 强 | ~1s (depth 4-6) | 中等 |
| MCTS | 很强 | 可控 | 较高 |
| 神经网络 (AlphaZero) | 最强 | 依赖 GPU | 很高 |

**选择 α-β + 启发式：** 对于 9×9 棋盘，depth 4 的 α-β 搜索已经足够强，普通人类很难赢。实现成本可控。

**搜索优化措施：**
- 只搜索已有棋子周围 2 格内的空位（邻域裁剪）
- 按启发式评分排序候选位置（提高 α-β 剪枝效率）
- 搜索宽度限制 10-15 个候选

#### 操控: 学习策略 vs 纯运动学

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| 纯 IK + 轨迹规划 | 确定性、无需训练 | 需手工调参接近角度/力度 | 简单取放 |
| **ACT (学习策略)** | 从示教学习完整动作 | 需要采集数据 | 复杂/精细操控 |
| Diffusion Policy | 多模态动作分布 | 推理慢 | 任务歧义高 |

**选择 ACT 的理由：**
- 扁平棋子的抓取不是简单的 point-to-point IK 问题
- 需要学习：接近角度、下压力度、夹爪闭合时机、提升轨迹、放置时的松手时机
- LeRobot 原生支持 ACT + SO-100，有现成的 pick-place 范例
- 30-50 次示教数据量可接受

**IK fallback：** ACT 策略如果泛化效果不好，可以退回到 IK + 预编程轨迹方案。两者可以在同一框架下切换。

## 2. 硬件方案

### 2.1 机械臂: SO-100

| 参数 | 规格 |
|------|------|
| 自由度 | 6 DOF + 1 夹爪 |
| 驱动 | 6× Feetech STS3215 舵机 |
| 臂展 | ~35-40cm |
| 工作空间 | ~50×50×50cm |
| 精度 | ±1-2° 重复定位 |
| 夹爪 | 平行夹爪，行程约 0-115° |
| 控制频率 | 30+ Hz |
| 接口 | USB 串口 (Feetech 总线协议) |

**工作空间分析：**

```
          SO-100 底座
              │
    ┌─────────┼─────────┐
    │         │  (home)  │
    │   ┌─────┴─────┐   │
    │   │   棋盘     │   │
    │   │  9×9      │   │
    │   │  20×20cm  │   │
    │   └───────────┘   │
    │                    │
    │   ┌────────────┐   │
    │   │  棋子托盘   │   │
    │   └────────────┘   │
    │                    │
    └────────────────────┘
         ~40cm
```

棋盘 20×20cm + 托盘 ~25cm，总占用约 25×45cm，在 SO-100 工作空间内。
**关键约束：** 棋盘边角是最远的位置，需要确认机械臂在最大伸展时仍有足够的操控裕度。建议棋盘中心距底座 20-25cm。

### 2.2 棋盘与棋子

| 项目 | 规格 | 备注 |
|------|------|------|
| 棋盘 | 9×9，网格间距 22mm，总尺寸 ~176×176mm | 亚克力/木板，哑光表面 |
| 棋子直径 | 18-20mm | 不能超过网格间距 |
| 棋子厚度 | **8-10mm** | 关键参数，太薄 (<5mm) 夹不住 |
| 棋子材质 | 塑料/树脂，底部平整 | 避免磁性干扰 |
| 棋子边缘 | 建议有 1mm 唇边/倒角 | 方便夹爪侧面夹持 |
| 托盘 | 线性排列，41 个槽位，间距 24mm | 3D 打印，槽宽 = 棋子直径 + 1mm |

**棋子厚度是关键设计参数。** SO-100 平行夹爪从侧面夹持棋子，棋子太薄（如围棋子 ~5mm）会导致：
- 夹持面积不足，容易滑落
- 接近棋盘时夹爪可能碰到棋盘表面

建议使用 8-10mm 厚度的棋子，或定制 3D 打印棋子。

### 2.3 相机

| 参数 | 要求 |
|------|------|
| 类型 | USB 摄像头 (OpenCV 兼容) |
| 分辨率 | 640×480 @ 30fps |
| 安装 | 顶部俯拍，距棋盘 40-50cm |
| 视野 | 需覆盖完整棋盘 (≥20×20cm) |
| 固定 | 三脚架/鹅颈夹持 |

**相机安装要求：**
- 光轴尽量垂直于棋盘平面（减少透视畸变）
- 棋盘完整在视野内，留有边距
- 安装后不可移动（否则需重新标定）
- 机械臂回到 home 位时不遮挡相机视野

### 2.4 辅助设备

| 设备 | 用途 | 备注 |
|------|------|------|
| LED 灯带/环形灯 | 稳定照明 | 漫射，避免反光和阴影 |
| ArUco 标记 | 棋盘定位 | 打印 4×4_50 字典，30×30mm |
| SO-100 Leader 臂 | 示教数据采集 | 仅数据采集阶段需要 |

### 3.4 坐标系与标定

**三个坐标系：**

1. **像素坐标系 (u, v):** 相机图像，单位像素
2. **棋盘坐标系 (col, row):** 棋盘交叉点索引，原点左上角 (0,0)
3. **机器人坐标系 (x, y, z):** SO-100 基坐标系，单位 mm

**变换链：**
```
像素 ──H (3×3 单应性)──▶ 棋盘 ──T (4×4 刚体)──▶ 机器人
```

**标定方法：**

| 步骤 | 方法 | 输入 | 输出 |
|------|------|------|------|
| 像素→棋盘 | ArUco 检测 + `cv2.findHomography` | 4 个角点像素坐标 | H (3×3) |
| 棋盘→机器人 | 3D 刚体变换 (SVD) | 4 个点的 EE 坐标 | T (4×4) |
| 托盘标定 | 两点定义直线 | 首末槽位 EE 坐标 | 原点 + 方向向量 |

**标定精度要求：**
- 像素→棋盘：误差 < 2 像素（在棋子直径范围内即可正确分类）
- 棋盘→机器人：误差 < 3mm（结合 ACT 策略的视觉反馈修正）

### 3.5 ACT 策略设计

**核心问题：如何让一个策略处理所有棋盘位置？**

方案：将目标位置作为 `observation.environment_state` 输入策略网络。

```
观测输入:
  ├── observation.images.top     (640×480×3)   相机画面
  ├── observation.state          (6,)          关节角度
  └── observation.environment_state (2,)       目标位置 (归一化 x,y)

动作输出:
  └── action                     (7,)          6 关节 + 夹爪
```

ACT 内部处理：
1. ResNet18 编码相机画面 → 视觉特征
2. 关节状态 + environment_state → 线性投影
3. Transformer encoder 融合视觉和状态特征
4. Transformer decoder 输出 chunk_size 个动作

**训练数据策略：**
- 30-50 个 episode，每个约 1.5-2s (45-60 帧 @ 30fps)
- 覆盖棋盘 4 个象限 + 中心 + 边缘位置
- 每个区域 5-8 个示教
- 录制时注入归一化目标坐标 `(x, y) ∈ [-1, 1]²`

**训练超参数：**

| 参数 | 值 | 说明 |
|------|-----|------|
| chunk_size | 50 | ~1.7s @ 30fps |
| n_action_steps | 25 | 执行 25 步后重新推理 |
| backbone | ResNet18 | ImageNet 预训练 |
| dim_model | 512 | Transformer 维度 |
| n_encoder_layers | 4 | |
| n_decoder_layers | 1 | |
| use_vae | True | 动作分布建模 |
| batch_size | 8 | 受数据量限制 |
| lr | 1e-5 | |
| steps | 50k | ~4h on RTX4090 |

### 3.6 IK Fallback 方案

如果 ACT 策略效果不理想，退回到运动学控制：

```python
def pick_and_place_ik(self, pick_pos, place_pos):
    # 1. 移到 pick 上方 (预抓取位)
    self.move_to(pick_pos + [0, 0, 50])  # 上方 50mm
    # 2. 下降到抓取高度
    self.move_to(pick_pos + [0, 0, 5])   # 棋子顶面
    # 3. 闭合夹爪
    self.set_gripper(80)  # 80%
    time.sleep(0.3)
    # 4. 提升
    self.move_to(pick_pos + [0, 0, 80])
    # 5. 移到 place 上方
    self.move_to(place_pos + [0, 0, 50])
    # 6. 下降放置
    self.move_to(place_pos + [0, 0, 5])
    # 7. 松开夹爪
    self.set_gripper(0)
    time.sleep(0.3)
    # 8. 缩回
    self.move_to(place_pos + [0, 0, 80])
```

IK 通过 `placo` 库（LeRobot 内置）计算。需要手工调的参数：
- 接近高度 (50mm / 80mm)
- 抓取高度 (棋子顶面 + 偏移)
- 夹爪闭合百分比
- 各阶段等待时间

## 4. 技术难点与应对

### 4.1 扁平棋子抓取

**难点：** 棋子厚度仅 8-10mm，平行夹爪难以稳定夹持。

**应对措施（优先级排列）：**
1. **棋子设计**: 3D 打印棋子，底部加 1mm 凸缘，增加夹持面积
2. **夹爪改造**: 在夹爪内侧贴硅胶垫，增加摩擦力
3. **接近策略**: 以微小角度（~5°倾斜）接近棋子，而非完全垂直
4. **示教覆盖**: 数据采集时包含不同接近角度的示教

### 4.2 边缘位置精度

**难点：** 棋盘边角距机械臂底座最远，精度下降。

**应对措施：**
1. 棋盘摆放时尽量居中于工作空间
2. 边缘位置多采集示教数据（双倍于中心区域）
3. 设置 `max_relative_target` 限制最大关节速度
4. 验证环节：落子后拍照确认偏差，必要时重试

### 4.3 遮挡问题

**难点：** 机械臂在运动中遮挡相机对棋盘的视野。

**应对措施：**
1. **时序控制**: 所有拍照操作都在机械臂回到 home 位后进行
2. **home 位设计**: 选择不遮挡棋盘的安全位置（如侧面或后方）
3. **可选**: 使用两个相机（正上方 + 侧面），互不遮挡
